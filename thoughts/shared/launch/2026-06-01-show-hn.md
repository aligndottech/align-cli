## Long body

Hi HN, I'm Tom.

I kept hitting the same annoying thing with my coding agent. It knows the codebase cold, but
it has no idea why any of it is shaped the way it is, so every now and then it cheerfully
rewrites something we'd deliberately settled months ago. The reasoning was never in the code.
It was in a Slack thread, a Jira comment and a call nobody wrote up, and none of that is in
front of the agent when it edits the file.

Docs tell an agent how the code works. The code tells it what got built. Neither tells it why,
or what a change would quietly contradict. The why is readable, to be clear - an agent can
search Slack or Jira over MCP just fine - it's just scattered across a dozen tools in prose,
unlinked, so nothing tells it which decision still holds, which got reversed, or what a new
change would contradict.

Align is my attempt at pulling that into one place an agent can query. The CLI is MIT:

  npm i -g @aligndottech/cli
  align setup

What it does: you connect the tools you already use (read-only) and Align pulls out the
decisions - what was decided, why, and who by - then links them across tools. The links are
typed, so it knows this decision supersedes that one, or this change conflicts with something
settled six months ago. The graph is exposed over MCP, so Claude Code or Cursor can ask
"what's been decided about auth" before they start writing, and `align check` can flag a
conflict on a diff.

By default align setup gives you a personal cloud graph: free, your own, synced and backed up.
If you'd rather keep everything on your machine, align setup --local runs it fully offline, a
local embedding model and a SQLite graph, no account, nothing leaving the box. Either way the
CLI and the MCP server are MIT, and the connectors it wires up for you are read-only. The useful
part isn't the storage, it's the typed conflict and supersession edges across tools, landing in
the agent's loop instead of leaving it to re-derive the same thing every session.

It gets in front of the agent two ways. While it's still planning, the agent queries Align over
MCP for related and conflicting decisions, and the server instructions tell it to do that before
writing anything, so a conflict can surface before any code exists. But that's the agent choosing
to check, and it doesn't always remember, so align setup also wires deterministic hooks into the
repo. A PreToolUse hook inspects a proposed edit before it's written to the file and feeds back
any decision it would contradict; a PostToolUse one does the same just after the edit lands, as a
backstop. Both are non-blocking and fail open. The PreToolUse one is the bit I'd most like torn
apart, the point being a conflict reaches the agent before the change is written, not at PR
review three days later.

A few honest things, since it's HN:

- I can't yet prove it makes an agent measurably better. What it does give you is discrete,
loggable events ("an agent or engineer was about to do X, which conflicts with decision Y"),
which at least makes a before/after possible. That's the honest state of it.

- No background listening. The CLI connects read-only and reads the history you point it at.
If even a personal cloud graph is too much, --local keeps the lot on your machine.

- Open-core, plainly: the CLI and the MCP server are MIT, this repo. A personal graph is free
(cloud or fully local); the paid side is the shared graph across a whole team.

- It's early and it's just me, so the relationship typing gets things wrong. I'd rather hear
where.

Two things I'd genuinely value your read on:

- Is feeding a conflicting decision back through the hook useful, or just more noise in an
already-full context window? What would make it earn the tokens?

- For a solo dev, is a decision graph over your own history worth anything, or does this only
really pay off on a team where you can't just go and ask the person who decided, partly
because there are too many of them and half have left?

Repo and install are in the link. I'll be about in the comments. Tell me what's rubbish.

Cheers,
Tom

## Comment replies (pre-written for the predicted objections)

Predicted from the Positioning Lab + Discovery Patterns tabs. Short, plain, in voice. Tweak per
the actual comment.

### "Isn't this just a vector DB of my docs / vectorised SQLite + an LLM call?"

The open-source CLI does store things as embeddings in SQLite, so for one repo on your laptop
it's close to that, and that's deliberate: free, private, no lock-in. The product isn't the
storage though. It's the typed edges across tools - this supersedes that, this conflicts with
that - and the fact they turn up in the agent's edit loop rather than in a search you have to
remember to run. The hosted side is where the heavier cross-tool detection runs. If the typed
graph doesn't beat a plain vector store for you, that's a real result and I'd want to hear it.

### "How do you know it actually makes agents better?"

Honestly, I can't prove the quality uplift yet, and I'm wary of anyone who claims they can. The
one thing this gives you that a wiki or a vector store doesn't is a discrete event every time
an agent or a person is about to do something that conflicts with a past decision. Those are
countable, so a before/after is at least possible. That's the bit I'm trying to measure
properly rather than assert.

### "This feels like a tool that listens to everything"

Fair, it's the first thing people raise. There's nothing sitting in the background monitoring
chat. You connect a source (read-only OAuth, or a local token) and it reads the history you
point it at, and you see what gets pulled in. If even that's too much, align setup --local
keeps the lot on your machine, SQLite and an on-device model, no account, no data leaving the
box. The hosted side is tenant-isolated with a self-host option.

### "Read-only? Doesn't the team version post into Slack/Jira?"

Good catch, worth being precise. The CLI and the personal tier connect read-only, that's what
this post is about. The team product has a bot that can post a decision summary back, but only
into a thread someone explicitly tags it in, never off its own back. So on the personal/CLI
side nothing gets written anywhere, and on the team side the only writes are the ones a person
asks for in the moment.

### "Is it cloud or local?"

Both, your call. align setup defaults to a free personal cloud graph (synced, backed up). If
you'd rather nothing left your machine, align setup --local runs the whole thing offline on
SQLite with an on-device model. Same CLI, same MIT code, you just pick where the graph lives.

### "ADRs already solve this"

I'm not trying to replace ADRs, and if your team keeps them well I'd just import them too. The
gap I keep hitting is the decisions that never become an ADR, the 4pm Slack thread where
everyone agrees and moves on, the call made in a PR review. In my experience that's most of
them, and they're the ones an agent trips over later. ADRs also need a separate act of writing
them up, whereas this reads from where the decision already happened. Complementary, not a
replacement.

### "Why not just a CLAUDE.md / AGENTS.md / repo docs?"

A CLAUDE.md is great and I'm not suggesting you drop it. But it's one file, one repo, and you
maintain it by hand, so it goes stale and it can't see a decision made in Jira or contradicted
in Slack last month. This is meant to be the layer underneath: cross-tool, and it flags when
something conflicts rather than you having to remember to write it down. Keep the CLAUDE.md,
point this at the messier stuff around it.

### "Open-core usually means the free bit is a crippled demo"

Reasonable to be wary. Straight version: the CLI and the MCP server are MIT, that's this repo,
and local mode genuinely works with no cloud, it's not a teaser. The hosted graph and the
cross-tool detection at team scale are the paid side. I'm not gating the API or making the free
version pointless on its own. Tell me if it still feels off.

### "Solo founder - will this still be here in a year, and what about my data?"

Fair, it's just me right now. Two things that should help: the CLI and MCP server are MIT so
they don't disappear if I do, and you can run fully local and export your graph, so your data
isn't stuck on the hosted side.
